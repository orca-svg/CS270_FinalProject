## 1. README 실행 절차 정확화 (`67d0e45`)

- **원인:** README에 클론부터 실행까지의 단계가 흩어져 있었고, 카메라 오버레이가
  `c / f / w / x` 키를 표시했지만 실제로는 `q`(종료)만 동작했다. 사용자가 동작하지
  않는 키를 시도할 위험이 있었다.
- **변경:** 5단계 Quick Start, dry-run 사전 점검 단계, 수동 BLE 테스트 예상 출력,
  트러블슈팅 섹션을 추가했다. 오버레이 키 안내가 비활성이라는 주의 문구를 명시했다.
- **해결/효과:** 처음 받는 팀원도 순서대로 따라 실행할 수 있고, 잘못된 키 입력으로
  인한 혼란을 제거했다.

## 2. 손 제스처 → 포물선 자동조준 교체 (`1b16708`)

- **원인:** 사용자가 컨트롤러를 손 제스처(MediaPipe) 방식에서 **빨간 물체 추적 +
  포물선 궤적 예측 + 자동 발사** 방식으로 바꿨다.
- **변경:** HSV 색상 마스킹으로 표적을 잡고, 속도(1차) + 수직 가속도(2차)를 EMA로
  추정해 미래 탄착점을 예측하고, 예측 오차가 화면 중심 임계 내이면 자동 발사하도록
  컨트롤러를 재작성했다. README도 알고리즘 4단계 설명으로 갱신했다.
- **해결/효과:** "던진 물체를 요격"하는 C-RAM 스타일 데모의 기반이 마련됐다.

## 3. 고정 카메라·독립 모터(절대 각도) 제어 (`6e69ae7`) — ⚠️ 이후 대체됨

- **원인:** 기존 코드는 카메라가 포탑(팬/틸트)에 장착되어 함께 움직인다고 가정하여,
  픽셀 오차를 Hub가 누적하는 방식이었다. 그러나 실제 구성은 **웹캠 고정 + 모터 독립**
  이었다.
- **변경:** 픽셀 좌표를 절대 모터 각도 `[-100, +100]`로 직접 변환하는
  `pixel_to_motor_vals()`를 추가하고, Hub도 누적 대신 절대 각도를 직접 설정하도록
  바꿨다 (`pan_target = pan_val/100 * PAN_MAX`).
- **해결/효과:** 당시 고정 카메라 구조에 맞는 직접 매핑을 구현했다.
- **현재 상태:** **대체됨.** PR #1(`7b8a815`)에서 팀이 다시 오차 누적 방식
  (`PAN_SIGN`, `GAIN`)으로 통일하고, 표적 요격 로직을 별도 파일
  `balloon_intercept.py`로 분리했다. 따라서 현재 Hub 프로토콜은 **오차 누적**이며,
  절대 각도 변환은 사용되지 않는다. (퇴행 방지를 위해 명시적으로 기록.)
  → 픽셀↔각도의 정확한 대응은 향후 [`PREDICTION.md`](PREDICTION.md)와 연계한
  **카메라-포탑 캘리브레이션**(로드맵 항목)으로 해결할 계획이다.

## 4. README 설계 동기화 + 로드맵·팀 (`cc143bb`)

- **원인:** 코드가 자동조준으로 바뀌었는데 README는 옛 제스처 설명이 남아 있었고,
  진행 상황·다음 작업·팀 분담이 문서화되어 있지 않았다.
- **변경:** README를 당시 코드 기준으로 동기화하고, "진행 상황 / 로드맵 / 5인 팀
  작업 분담(기기 1대 제약 반영)" 섹션을 추가했다. 한국어 README도 전면 재작성했다.
- **해결/효과:** README가 단순 소개를 넘어 **팀 운영 문서** 역할을 시작했다.

## 5. 역할 확정 + 대시보드 (`e9926af`)

- **원인:** 팀이 5인 역할(P1~P5)을 확정했고, 진행 상황을 더 가시적으로 볼 표가
  필요했다.
- **변경:** 역할을 확정 표로 정리하고, 모듈별 상태(✅/🔜/⬜)와 담당자를 매핑한
  대시보드, 우선순위 할 일 표(기기 필요 여부 포함)를 추가했다.
- **해결/효과:** 누가 무엇을 맡고 어디까지 됐는지 한눈에 파악 가능해졌다.

## 6. PR #1 — Pybricks BLE 저장소 재구성 (`7b8a815`)

- **원인:** 공유 저장소를 Pybricks BLE 직결 방향으로 명확히 하고, 코드 구조를
  정돈할 필요가 있었다.
- **변경:**
  - 공유 BLE 클라이언트 `pybricks_ble.py` 분리 (스캔/재연결/readiness/진단)
  - 자동조준 로직을 `balloon_intercept.py`로 분리 (HSV + 포물선 + 자동 발사)
  - 손 제스처 컨트롤러 `gesture_bt_controller.py` 유지
  - Hub 파서 자가 복구(디싱크 재동기화), stdin flush, 크래시 가시화
    (`ERR_*`, `FATAL`, `BTN_STOP`), 발사 스냅샷 로그(`SHOT_START/RELEASE/DONE`) 추가
  - `docs/`에 ARCHITECTURE / PROTOCOL / STATE_MACHINES / PREDICTION 추가
  - Hub 제어를 **오차 누적**(`PAN_SIGN`, `GAIN`)으로 통일 → 항목 3을 대체
- **해결/효과:** 모듈 책임이 분리되고, BLE 안정성·진단 가시성이 크게 향상됐다.

## 7. README 가독성 재작성 + dry-run drift 수정 (`b256832`)

- **원인 1 (가독성):** 재구성 후 README가 길고 밀도가 높아 빠르게 훑기 어려웠다.
- **원인 2 (drift):** 대시보드가 "camera-only / no-BLE 모드"를 미완료로 표기했으나,
  실제로는 `gesture_bt_controller.py`와 `balloon_intercept.py` **둘 다 이미
  `--dry-run`을 지원**했다 (코드와 문서 불일치).
- **변경:**
  - 상단 1줄 아키텍처 다이어그램 + "어떤 스크립트를 실행하나?" 진입점 표 추가
  - 이모지 섹션 헤더, ✅/🔜/⬜ 아이콘 대시보드로 스캔 용이성 강화
  - 심화 내용은 중복 대신 `docs/`로 링크 (progressive disclosure)
  - `--dry-run`을 ✅ 완료로 정정하고 상단에 강조; 남은 작업을 "결정적 녹화 영상
    리플레이 하네스"로 재정의
  - 프로토콜·옵션 표를 실제 CLI 인자와 교차 검증 (`--fire-cooldown` 등 반영)
- **해결/효과:** README가 실제 코드와 일치하고, 신규 진입자가 5초 안에 "무엇을
  실행할지" 판단할 수 있게 됐다.

## 핵심 교훈

- **문서는 코드 기준으로 검증한다.** 항목 7의 dry-run drift처럼, 표기와 실제 기능이
  어긋나면 잘못된 우선순위로 이어진다.
- **대체된 결정도 기록한다.** 항목 3의 절대 각도 방식은 폐기됐지만, 기록이 없으면
  다음 사람이 같은 변경을 반복하거나 혼란을 겪는다.
- **기기 1대 제약을 설계에 반영한다.** `--dry-run` 덕분에 5명 중 3명이 로봇 없이
  병렬 작업할 수 있다 — 이것이 팀 처리량의 핵심이다.
